### 5.1.4.16 网络诊断报文

网络诊断报文（`MMeDiagnose`）为各厂商非标准的自定义报文，用于网络故障定位，协议只统一了报文前导字段，不统一厂家自定义内容格式。

#### 表 135 网络诊断报文格式

| 字段 | 字节号 | 字段大小（比特） |
| --- | --- | --- |
| 芯片厂商 ID | 0-1 | 16 |
| 厂家自定义 | 可变长 | 可变长 |

#### 表 136 芯片厂商 ID 定义

| 芯片厂商 | ID 值 |
| --- | --- |
| `0x0000` | 保留 |
| `0x0001` | HS |
| `0x0002` | ES |
| `0x0003` | TC |
| `0x0004` | LH |
| `0x0005` | HT |
| `0x0006` | RS |
| `0x0007` | SW |
| `0x0008` | SC |
| `0x0009-0xFFFF` | 待扩展 |

## 实现说明

- `16_MMeDiagnose.h/.c` 按“固定 2 字节芯片厂商 ID + 原始厂家自定义区”实现了通用编解码。
- 厂家自定义区保持原始字节视图，不擅自拆解为更深层的厂商私有结构。
- `validate` 当前做三类校验：
- `芯片厂商 ID` 不能为保留值 `0x0000`
- 厂家自定义区长度不能超过 `65533` 字节
- 当厂家自定义区长度非 `0` 时，数据指针不能为空
- 解码成功后，`Message->VendorDefinedData` 直接指向输入缓冲区中的厂家自定义区，不额外拷贝。
- 本实现允许厂家自定义区长度为 `0`，即只有统一的 `2` 字节厂商 ID 头。
  这是因为规范对厂家自定义区未规定最小长度，同时 1.0 旧版结构体也只统一到 `ChipManufID` 字段。
- `0x0009-0xFFFF` 在规范中标记为“待扩展”，因此本实现允许该区间作为扩展厂商 ID 透传，只对 `0x0000` 做保留值拦截。

## 与 1.0 版本对比

1.0 对比基线为 `doc/南网hplc+hrf双模模块1.0标准规范/2、计量自动化系统双模通信技术规范/《南方电网计量自动化系统双模通信技术规范 第2部分：无线通信接入协议》（报批稿）.pdf` 的 `8.1.4.16 网络诊断报文`（表 `210`、表 `211`）；2.0 对应当前文件整理的 `5.1.4.16 网络诊断报文`（表 `135`、表 `136`）。

两版报文的整体帧布局未变化，均只统一定义前导 `2` 字节芯片厂商 ID 和后续变长厂家自定义区，厂家自定义内容仍由各厂商自行约定。

| 对比项 | 1.0 版 | 2.0 版 | 结论 |
| --- | --- | --- | --- |
| 章节定位 | `8.1.4.16`，表 `210/211` | `5.1.4.16`，表 `135/136` | 仅章节编号和表号调整 |
| 报文固定部分 | `0-1` 字节为芯片厂商 ID | `0-1` 字节为芯片厂商 ID | 固定头长度一致 |
| 厂家自定义区 | `2` 字节后为可变长厂家自定义区 | `2` 字节后为可变长厂家自定义区 | 载荷组织方式一致 |
| 芯片厂商 ID 取值 | `0x0000` 保留，`0x0001-0x0008` 为 `HS/ES/TC/LH/HT/RS/SW/SC`，`0x0009-0xFFFF` 待扩展 | 与 1.0 相同 | 枚举定义一致 |
| 报文语义 | 用于网络故障定位，厂家自定义内容不做统一 | 用于网络故障定位，协议只统一前导字段 | 表述收敛，但语义不变 |

因此，`2.0` 对 `MMeDiagnose` 属于沿用 `1.0` 定义的兼容性延续，并非重新设计，实质差异主要体现在章节重编排和文字表述整理。
