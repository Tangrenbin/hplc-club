## 实现说明

- `21_MMeRuSnrInfoNotify.h/.c` 实现了主报文固定头、邻居站点位图和原始 `RU_SNR` 信息区的完整编解码。
- 主报文中的前 `3` 个字节使用连续 `24 bit` 打包两个 `12 bit` 的 `TEI`：
- 低 `12 bit` 为 `LocalTei`
- 高 `12 bit` 为 `NeighborStartTei`
- `Byte3` 使用 `MMeRuSnrInfoNotifyByte3_t` 表示：
- `bit[2:0]` 为 `RuGroupId`
- `bit[7:3]` 为 `Reserved`
- `validate` 会校验：
- `LocalTei` 与 `NeighborStartTei` 都不超过 `0x0fff`
- `RuGroupId` 不超过 `0x07`，且保留位必须为 `0`
- 邻居位图实际置位个数必须等于 `NotifiedNeighborCount`
- 当 `RuGroupId == 0` 时，`NeighborRuSnrInfoListLength` 必须等于 `NotifiedNeighborCount * 4`
- 代码另外提供了 `RU 分组 ID 0` 的单邻居 `4` 字节辅助编解码接口：
- `mme_ru_snr_info_notify_ru_group0_info_encode`
- `mme_ru_snr_info_notify_ru_group0_info_decode`
- `RU0_SNR ~ RU5_SNR` 都按 `5 bit` 量化值处理，编码时保留位强制写 `0`，解码时若保留位非 `0` 则失败
- 对 `RuGroupId != 0` 的情况，当前实现只保留主报文中的原始信息区，不继续猜测更深层结构

## 校准说明

- 原 PDF 为扫描版，`pdftotext` 对第 `68` 页几乎只能提取水印；本文件基于页图 OCR 结果人工校准。
- 第 `68` 页顶部仍残留上一节 `5.1.4.20 Bitloading训练结果更新确认` 的表 `157` 尾部；`5.1.4.21` 从该页中部开始。
- 表 `158` 中 `本站点 TEI` / `邻居站点起始 TEI` 的 `12 bit + 12 bit` 打包方式，与同页上方表 `157` 的双 `TEI` 复用布局一致，因此本实现沿用相同的 `3` 字节打包方式。
- 表 `159` 的 OCR 存在跨行断裂，例如 `RU1_SNR`、`RU3_SNR`、`RU4_SNR` 的比特位被拆散到两行；本实现按连续 `30 bit` 的 `6` 个 `5 bit` 字段重新校准为：
- `RU0` 占 bit `0-4`
- `RU1` 占 bit `5-9`
- `RU2` 占 bit `10-14`
- `RU3` 占 bit `15-19`
- `RU4` 占 bit `20-24`
- `RU5` 占 bit `25-29`
- bit `30-31` 为保留
- 本仓库与 SDK 中当前没有现成的 `MMeRuSnrInfoNotify` 枚举或结构体定义，因此消息类型 `0x0074` 属于按章节顺序推定值：
- `5.1.4.17` 无线信道冲突上报报文在 SDK `mmsg.h` 中已确认是 `0x0070`
- `5.1.4.18` 无线发现列表报文在仓库中已按顺序推定为 `0x0071`
- `5.1.4.20` Bitloading训练结果更新确认在仓库中已按顺序推定为 `0x0073`
- 因此本次将 `5.1.4.21` 顺序对应推定为 `0x0074`
