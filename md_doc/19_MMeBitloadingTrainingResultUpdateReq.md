### 5.1.4.19 Bitloading训练结果更新请求

规范原文说明：

接收端生成训练结果后，通过 Bitloading训练结果更新请求将训练结果反馈给发端站点。

#### 表 153 Bitloading训练结果更新请求数据格式

| 字段 | 字节号 | 字段大小(比特) | 备注 |
| --- | --- | --- | --- |
| 源 TEI | 0-1 | 12 | - |
| 目的 TEI | 1-2 | 12 | - |
| 流数 | 3 | 2 | `0`：单流；`1`：双流 |
| 子载波组大小 | 3 | 2 | 结合 PHY 头字段校准为 `0/1/2/3` 分别表示每组 `1/2/4/8` 个子载波 |
| 保留 | 3 | 4 | - |
| 截止载波 | 4-5 | 9 | 截止载波后的载波不承载数据；规范示例说明“全带宽都可用，则填 `490`” |
| 保留 | 5 | 7 | - |
| 比特加载表长度 | 6-7 | 16 | 单位字节 |
| 比特加载表 | 8-... | 可变长 | 表示有效载波频段范围内每个子载波组中的每个子载波所承载的比特数 |

#### 表 154 单流比特加载表数据格式

- 单流场景下，每个子载波组占 `3 bit`
- 每 `8` 个子载波组占 `3` 字节

#### 表 155 双流比特加载表数据格式

- 双流场景下，流 `0` 与流 `1` 各自按单流格式编码
- 规范说明按“`8` 组流 `0` + `8` 组流 `1`”的方式顺序排列

#### 表 156 比特承载信息定义

- `0`：不承载数据
- `1`：承载 `1 bit`，对应 `BPSK`
- `2`：承载 `2 bit`，对应 `QPSK`
- `4`：承载 `4 bit`，对应 `16QAM`
- `6`：承载 `6 bit`，对应 `64QAM`

## 实现说明

- `19_MMeBitloadingTrainingResultUpdateReq.h/.c` 实现了固定 `8` 字节头和原始比特加载表的完整编解码。
- 前 `3` 个字节沿用 `12 bit SourceTei + 12 bit DestinationTei` 的连续 `24 bit` 打包方式，与已实现的 `MMeBitloadingTrainingResultUpdateCnf` 保持一致。
- 字节 `3` 解析为：
- `bit[1:0]`：`FlowCount`
- `bit[3:2]`：`SubCarrierGroupSize`
- `bit[7:4]`：`ReservedByte3`
- 字节 `4-5` 解析为：
- `byte4 + byte5 bit[0]`：`9 bit CutoffCarrier`
- `byte5 bit[7:1]`：`ReservedByte5`
- 比特加载表当前保持原始字节视图，不继续拆为表 `154/155` 的 3-bit 组级结构。
- 解码成功后，`Message->BitloadingTable` 直接指向输入缓冲区中的原始表数据，不额外拷贝。

## 校准说明

- 原 PDF 为扫描版，`pdftotext` 对第 `66` 页几乎只抽出水印；本实现以页图 OCR 为主，结合协议上下文人工校准。
- `源 TEI`、`目的 TEI`、`流数`、`截止载波`、`比特加载表长度` 以及“`0` 单流 / `1` 双流”的字段语义来自第 `66` 页表 `153`。
- `子载波组大小` 的取值含义使用 `h2-link-v2-sdk/SDK/src/include/phy/framectrl.h` 中 `sc_grp_sz` 定义交叉确认：
- `0`：`1` 个子载波
- `1`：`2` 个子载波
- `2`：`4` 个子载波
- `3`：`8` 个子载波
- 字节 `3` 高 `4 bit` 保留、字节 `5` 高 `7 bit` 保留的布局，来自“表 `153` 固定头总长为 `8` 字节”和 OCR 可辨字段的联合推断；当前仓库与 SDK 中尚无现成结构体可直接核对。
- 消息类型 `0x0072` 属于按章节顺序推定值，不是 SDK 中现成枚举值：
- `5.1.4.17` 无线信道冲突上报报文在 SDK `mmsg.h` 中已确认是 `0x0070`
- 仓库现有 `5.1.4.18` 无线发现列表报文实现已按顺序推定为 `0x0071`
- 因此本次将 `5.1.4.19` / `5.1.4.20` 顺序对应推定为 `0x0072` / `0x0073`
- 若后续 SDK 或规范修订稿补充了正式枚举与结构体，应以正式定义回填类型值和位布局说明
