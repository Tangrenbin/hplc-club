# MMeChangeProxyBitMapCnf

## 协议章节

### 消息标识

- 管理消息名称：代理变更请求确认报文（位图版）（`MMeChangeProxyBitMapCnf`）
- 管理消息类型：`0x003B`
- 来源文档：`doc/南网hplc+hrf双模模块2.0标准规范/通感一体化低压电力线宽带载波通信规约/4-通感一体化低压电力线宽带载波通信规约 第4部分 数据链路层通信协议 （初稿).pdf`
- 对应 PDF 物理页：54-55
- 对应正文页码：52-53

### 5.1.4.7 代理变更请求确认报文（位图版）

代理变更请求确认报文（位图版）（`MMeChangeProxyBitMapCnf`）格式的定义如表 `111` 所示。

#### 表 111 代理变更请求确认报文（位图版）格式

| 字段 | 字节号 | 字段大小（字节/比特） | 说明 |
| --- | --- | --- | --- |
| 结果 | 0-3 | 4 字节 | 见表 `112` |
| 站点 TEI | 4-5 | 2 字节 | 申请代理变更的站点 TEI |
| 代理 TEI | 6-7 | 2 字节 | 申请代理变更站点的新代理 TEI |
| 组网序列号 | 8 | 1 字节 | 代理变更请求发起时的组网序列号 |
| 子站点位图 | 9-138 | 130 字节 | 按位图表示申请代理变更站点的所有子站点 TEI |
| 链路类型 | 139 bit0 | 1 bit | `0` 表示高速载波链路，`1` 表示无线链路 |
| 保留 | 139 bit[7:1] | 7 bit | 保留 |
| 端到端序列号 | 140-143 | 4 字节 | 原代理变更请求中的端到端报文序列号 |
| 路径序号 | 144-147 | 4 字节 | 当前路径通知序列号 |

#### 表 112 代理变更结果

| 值 | 定义 |
| --- | --- |
| `0x00000000` | 表示代理变更成功 |

#### 5.1.4.7.2 站点 TEI

- 表示申请代理变更的站点的 TEI。

#### 5.1.4.7.3 代理 TEI

- 表示申请代理变更站点的新代理站点的 TEI。

#### 5.1.4.7.4 组网序列号

- 表示代理变更请求发起时，该网络的组网序列号。

#### 5.1.4.7.5 子站点位图

- 采用位图表示代理变更请求发起站点的所有子站点 TEI。
- 根据 TEI 大小在位图中相应的位置填写标志，若比特位值为 `1`，表示对应 TEI 有效。
- 规约示例：
- 第 `0` 字节的第 `1` 比特为 `1`，表示 `TEI=1` 的站点是该站点的子站点。
- 第 `1` 字节的第 `0` 比特为 `1`，表示 `TEI=8` 的站点是该站点的子站点。

#### 5.1.4.7.6 端到端序列号

- 表示端到端的报文序列号。
- 请求代理变更的站点在产生代理变更请求报文时，需要获取一个序列号。
- CCO 在确认代理变更时，需要在代理变更请求确认报文中携带代理变更请求报文中的端到端报文序列号。

#### 5.1.4.7.7 路径序号

- 表示路径通知序列号。
- 每次发送代理变更请求确认报文时，CCO 会获取一个递增的路径通知序列号。
- 代理站点或者 STA 站点在刷新路由表项时，需要判断路径序列号是否为最新值。

## 实现说明

- `07_MMeChangeProxyBitMapCnf.h/.c` 按表 `111` 实现为固定 `148` 字节报文。
- `子站点位图` 使用固定 `130` 字节数组保存，并提供：
- `mme_change_proxy_bitmap_cnf_set_sub_station_tei`
- `mme_change_proxy_bitmap_cnf_get_sub_station_tei`
- `validate` 当前会校验：
- `结果` 固定为 `0x00000000`
- `链路类型` 只允许 `0x00` 和 `0x01`
- 字节 `139` 的 `7` 个保留位必须为 `0`
- 编码和解码均采用小端字节序，和仓库内现有管理消息保持一致。

## 与 1.0 版本对比

对比基准为 `doc/南网hplc+hrf双模模块1.0标准规范/2、计量自动化系统双模通信技术规范/《南方电网计量自动化系统双模通信技术规范 第2部分：无线通信接入协议》（报批稿）.pdf` 的 `8.1.4.7 代理变更请求确认报文（位图版）`（表 `189` 至表 `190`）。

从报文整体布局看，`2.0` 版 `MMeChangeProxyBitMapCnf` 与 `1.0` 版保持实质一致，固定长度仍为 `148` 字节，`结果`、`站点 TEI`、`代理 TEI`、`组网序列号`、`130` 字节 `子站点位图`、字节 `139` 的 `链路类型 + 保留位`、`端到端序列号`、`路径序号` 的排列和语义均未见变化；主要区别是规范章节从 `8.1.4.7` 迁移到 `5.1.4.7`，表号由 `189/190` 调整为 `111/112`。

| 对比项 | 1.0 版 | 2.0 版 | 结论 |
| --- | --- | --- | --- |
| 章节归属 | `8.1.4.7`，表 `189-190`，归属于 `1.0` 双模无线通信接入协议 | `5.1.4.7`，表 `111-112`，归属于 `2.0` 数据链路层通信协议 | 主要是规范迁移与重编号 |
| 报文总长与字段顺序 | 固定 `148` 字节，字段顺序为 `结果`、`站点 TEI`、`代理 TEI`、`组网序列号`、`子站点位图`、`链路类型/保留`、`端到端序列号`、`路径序号` | 与 `1.0` 相同 | 外层布局未变 |
| 子站点表达方式 | 使用固定 `130` 字节位图，按 TEI 对应比特位标记子站点 | 与 `1.0` 相同 | 位图编码规则未变 |
| 字节 `139` 位域 | `bit0` 为 `链路类型`，其余 `7 bit` 保留 | 与 `1.0` 相同 | 位域定义未变 |
| 结果码定义 | 表 `190` 仅定义 `0x0` 表示变更成功 | 表 `112` 仅定义 `0x00000000` 表示变更成功 | 仅写法补齐为 `32 bit` 表示，语义未变 |

整体上，`2.0` 版 `MMeChangeProxyBitMapCnf` 可以视为对 `1.0` 位图版代理变更确认报文的直接承接，属于规范迁移和表号调整，而不是报文格式重设计。

## 校准说明

- 原 PDF 为扫描版，`pdftotext` 只能提取到水印；表 `111/112` 依据物理页 `54-55` 页图人工校准。
- 字段顺序、固定长度 `148` 字节、`130` 字节子站点位图以及尾部 `链路类型 + 端到端序列号 + 路径序号` 的布局，使用 `h2-link-v2-sdk/SDK/src/spg/mac/mmsg.h` 中的 `MMeChangeProxyBitMapCnf_t` 做了交叉确认。
- 规约正文没有单独展开字节 `139 bit0` 的链路类型取值说明，本实现沿用 `5.1.4.6` 普通确认报文中的同名字段语义，并与 SDK 中的 `if_type` 位保持一致，按 `0=高速载波链路、1=无线链路` 处理。
