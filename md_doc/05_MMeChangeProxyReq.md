### 5.1.4.5 代理变更请求报文

代理变更请求报文（`MMeChangeProxyReq`）格式的定义如表 104 所示。

#### 表 104 代理变更请求报文格式

| 字段 | 字节号 | 字段大小（字节/比特） | 说明 |
| --- | --- | --- | --- |
| 站点 TEI | 0-1 | 2 字节 | 申请进行代理变更站点的设备标识 |
| 新代理 TEI[0] | 2-3 | 2 字节 | 候选新代理 0 的 TEI |
| 新代理 TEI[1] | 4-5 | 2 字节 | 候选新代理 1 的 TEI |
| 新代理 TEI[2] | 6-7 | 2 字节 | 候选新代理 2 的 TEI |
| 新代理 TEI[3] | 8-9 | 2 字节 | 候选新代理 3 的 TEI |
| 新代理 TEI[4] | 10-11 | 2 字节 | 候选新代理 4 的 TEI |
| 旧代理 TEI | 12-13 | 2 字节 | 申请代理变更站点的原代理站点 TEI |
| 代理类型 | 14 | 1 字节 | 原代理类型，定义见表 105 |
| 原因 | 15 | 1 字节 | 代理变更原因，定义见表 106 |
| 站点相线[0] | 16 | 1 字节 | 最低字节，存放评估出的所属相线 |
| 站点相线[1] | 17 | 1 字节 | 备选相线 1 |
| 站点相线[2] | 18 | 1 字节 | 备选相线 2 |
| 新代理 0 链路类型 | 19 | 1 bit | 候选新代理 0 的链路类型 |
| 新代理 1 链路类型 | 19 | 1 bit | 候选新代理 1 的链路类型 |
| 新代理 2 链路类型 | 19 | 1 bit | 候选新代理 2 的链路类型 |
| 新代理 3 链路类型 | 19 | 1 bit | 候选新代理 3 的链路类型 |
| 新代理 4 链路类型 | 19 | 1 bit | 候选新代理 4 的链路类型 |
| 保留 | 19 | 3 bit | 保留 |
| 端到端序列号 | 20-23 | 4 字节 | 端到端报文序列号 |
| 组网序列号 | 24 | 1 字节 | 代理变更请求发起时的组网序列号 |
| 保留 | 25-39 | 15 字节 | 保留 |

#### 5.1.4.5.1 站点 TEI

- 表示申请进行代理变更站点的设备标识。

#### 5.1.4.5.2 新代理 TEI

- 表示候选代理站点的设备标识。
- 每 2 个字节携带一个候选代理 TEI，最多支持 5 个。
- 由发起代理变更请求的站点向 CCO 提出候选代理站点，CCO 会在候选代理中指定该站点的新代理。

#### 5.1.4.5.3 旧代理 TEI

- 表示申请代理变更站点的原代理站点的设备标识。

#### 5.1.4.5.4 代理类型

- 表示申请代理变更站点的原代理类型，定义如表 105 所示。

#### 表 105 代理类型

| 值 | 定义 |
| --- | --- |
| `0x01` | 保留 |
| `0x02` | 表示动态代理 |

#### 5.1.4.5.5 原因

- 表示站点发起代理变更的原因，定义如表 106 所示。

#### 表 106 代理变更原因

| 值 | 定义 |
| --- | --- |
| `0x01` | 表示周期代理变更 |
| `0x02` | 表示快速代理变更（周期内快速变更） |
| 其他值 | 保留 |

#### 5.1.4.5.6 站点相线

- 表示发起代理变更请求站点的相线信息。
- 最低字节存放评估出的所属相线，其他字节依次填入可能的备选相位。
- 相线值定义如表 107 所示。

#### 表 107 相线评估信息

| 值 | 定义 |
| --- | --- |
| `0` | 表示未知相线 |
| `1` | 表示 A 相线 |
| `2` | 表示 B 相线 |
| `3` | 表示 C 相线 |

#### 5.1.4.5.7 端到端序列号

- 表示端到端的报文序列号。
- 请求代理变更的站点在产生代理变更请求报文时，需要获取一个序列号。
- CCO 在代理变更确认报文中，需要携带本请求报文中的端到端报文序列号。
- 该序列号由发起请求的站点维护。

## 实现说明

- `05_MMeChangeProxyReq.h/.c` 按表 104 实现为固定 40 字节报文。
- 字节 `19` 实现为 `5` 个候选代理链路类型位和 `3` 个保留位。
- `validate` 会校验：
  - `代理类型` 只允许 `0x01` 和 `0x02`
  - `原因` 只允许 `0x01` 和 `0x02`
  - `站点相线[0..2]` 只允许 `0..3`
  - `字节 19` 的保留位为 `0`
  - `字节 25-39` 的保留区全部为 `0`

## 与 1.0 版本对比

对比基准为 `doc/南网hplc+hrf双模模块1.0标准规范/2、计量自动化系统双模通信技术规范/《南方电网计量自动化系统双模通信技术规范 第2部分：无线通信接入协议》（报批稿）.pdf` 的 `8.1.4.5 代理变更请求报文`（表 `182` 至表 `185`）。

从报文整体布局看，`2.0` 版 `MMeChangeProxyReq` 仍保持 `1.0` 的固定 `40` 字节结构，`站点 TEI`、`5` 个 `新代理 TEI`、`旧代理 TEI`、`代理类型`、`原因`、`站点相线`、候选代理链路类型位、`端到端序列号`、`组网序列号` 以及尾部 `15` 字节保留区的位置均未变化；当前可确认的差异主要是规范章节和表号迁移，未见新增字段或字段语义重定义。

| 对比项 | 1.0 版 | 2.0 版 | 结论 |
| --- | --- | --- | --- |
| 章节归属 | `8.1.4.5`，表 `182-185`，归属于 `1.0` 双模无线通信接入协议 | `5.1.4.5`，表 `104-107`，归属于 `2.0` 数据链路层通信协议 | 规范迁移并重编号 |
| 报文总长与主体骨架 | 固定 `40` 字节，字节 `0-24` 为有效字段，字节 `25-39` 为保留 | 与 `1.0` 相同，仍为固定 `40` 字节 | 外层布局未变 |
| 代理候选字段 | `站点 TEI`、`新代理 TEI0-4`、`旧代理 TEI` 顺序排列，候选代理最多 `5` 个 | 与 `1.0` 相同，字段顺序和数量未变 | 代理候选描述未变 |
| 字节 `19` 位域 | `5` 个候选新代理链路类型位加 `3 bit` 保留位 | 与 `1.0` 相同，仍用于描述 `5` 个候选代理的链路类型 | 位域定义未变 |
| 枚举与序列号语义 | `代理类型` 仅定义动态代理，`原因` 定义周期/快速两类，并携带 `端到端序列号`、`组网序列号` | 当前 `2.0` 文件的已校准内容与 `1.0` 一致，未见新增有效取值或新增序列字段 | 业务语义延续 `1.0` |

整体上，`2.0` 版 `MMeChangeProxyReq` 可以视为对 `1.0` 定义的直接承接，属于规范迁移与重编号，而不是报文格式重设计。

## 校准说明

- 原 PDF 为扫描版，`pdftotext` 基本只能提取到水印，需要结合页面图像 OCR 人工校准。
- 表 104 中 `新代理 TEI` 行在 OCR 里存在错位，结合 `h2-link-v2-sdk/SDK/src/spg/mac/mmsg.h` 的 `MMeChangeProxyReq_t`，可确认该字段是连续 `5` 个 `uint16_t`，对应字节 `2-11`。
- 表 104 中字节 `24` 的字段在 OCR 小节标题里没有单独展开，但表格中明确存在“组网序列号”，且与 SDK 结构中的 `networking` 字节位置一致；本实现按仓库现有命名统一建模为 `FormationSequenceNumber`。
- 表 106 的保留取值范围 OCR 质量不足，只能稳定识别 `0x01` 和 `0x02` 两个有效值；本实现将其他值统一视为非法输入。
