### 5.1.4.11 发现列表报文

发现列表报文（`MMeDiscoverNodeList`）由固定头、上行路由条目区、固定 `128` 字节发现站点位图区和接收发现列表信息区组成。

#### 表 119 发现列表报文固定头

| 字段 | 字节号 | 字段大小（比特） | 说明 |
| --- | --- | --- | --- |
| TEI | 0-1 | 16 | 发送本报文的站点 TEI |
| 角色 | 2 | 8 | 站点角色 |
| 层级 | 3 | 8 | 网络层级 |
| MAC 地址 | 4-9 | 48 | 站点 MAC 地址 |
| 代理 TEI | 10-11 | 16 | 代理站点 TEI |
| PLC 能力指示 | 12 bit[1:0] | 2 | `0` 表示仅支持 `BPLC`，`1` 表示支持 `BPLC + ISAC-PLC` |
| 通道数 | 12 bit[2] | 1 | `0` 单通道，`1` 多通道 |
| 保留 | 12 bit[30:3] | 28 | 保留，编码时置 `0` |
| 与代理站点通信成功率计算完成标志 | 15 bit[7] / bit[31] | 1 | `1` 表示已经完成成功率计算 |
| 与代理站点通信成功率 | 16-19 | 32 | 与代理站点之间的上下行通信成功率 |
| 与代理站点下行通信成功率 | 20-23 | 32 | 接收代理站点下行报文的成功率 |
| 站点总数 | 24-25 | 16 | 本报文携带发现站点信息的站点数量 |
| 发送发现列表报文个数 | 26-27 | 16 | 上个路由周期内发送的发现列表报文总数 |
| 上行路由条目总数 | 28-29 | 16 | 到达 CCO 的上行路由表项数目，最大支持 `4` 条 |
| 接收发现列表信息条目长度 | 30 | 8 | 固定值 `8`，表示每个接收计数条目占 `1` 字节 |
| 保留 | 31-32 | 16 | 保留，编码时置 `0` |
| 路由周期到期剩余时间 | 33-34 | 16 | 距离当前路由周期到期的剩余时间，单位秒 |
| 相线 | 35 | 8 | 第一、第二、第三相线评估结果 |
| 最小通信成功率 | 36 | 8 | 到 CCO 路径上的最小通信成功率 |
| 与代理的 RU_SNR 评估信息 | 37-40 | 32 | 本实现按原始 4 字节保留 |
| 保留 | 41 | 8 | 保留，编码时置 `0` |

#### 5.1.4.11.16 相线

- 字节 `35` 的位域布局如下：
  - bit[1:0]：第三相线
  - bit[3:2]：第二相线
  - bit[5:4]：第一相线
  - bit[7:6]：保留
- 相线取值沿用数据链路层已有定义：`0` 未知，`1` A 相，`2` B 相，`3` C 相。

#### 5.1.4.11.19 上行路由条目信息

- 紧跟固定头之后，为可变长度区。
- 每个路由条目固定 `3` 字节：
  - 字节 `0-1`：下一跳站点 `TEI`
  - 字节 `2`：路由类型
- 路由类型按表 125 实现：
  - `0x00`：错误的路由类型
  - `0x01`：同级备份路由类型
  - `0x02`：上级备份路由类型
  - `0x03`：代理主路径路由类型
  - `0x04`：上上级路由类型
- 本实现要求：当 `上行路由条目总数 > 0` 时，条目中至少包含一条 `代理主路径路由类型`。

#### 5.1.4.11.20 发现站点列表位图

- 位图长度固定为 `128` 字节。
- 位图中比特位为 `1` 表示对应 `TEI` 站点可被发现，并且本报文携带了对应站点的接收发现列表信息。
- 位图映射规则与规范示例一致：
  - 第 `0` 字节 bit1 为 `1`，表示 `TEI=1`
  - 第 `1` 字节 bit0 为 `1`，表示 `TEI=8`
- 代码提供 `mme_discover_node_list_set_discovered_tei` 和 `mme_discover_node_list_get_discovered_tei` 两个辅助接口。

#### 5.1.4.11.21 接收发现列表信息

- 发现站点位图之后是接收发现列表信息区。
- 每个条目固定 `1` 字节，对应一个被位图置位的站点。
- 本实现将该区域保留为 `ReceivedDiscoverListInfo` 指针，不额外拆成更深层结构。
- `StationCount` 必须与位图中实际置位个数一致，编码与解码都会校验该约束。

## 实现说明

- `11_MMeDiscoverNodeList.h/.c` 已实现固定头字段、上行路由条目、固定 `128` 字节发现站点位图和接收发现列表信息区的完整编解码。
- `validate` 当前会校验：
  - 角色、PLC 能力、通道数、相线、路由类型取值合法；
  - 三个通信成功率字段都处于 `0-100` 范围；
  - `接收发现列表信息条目长度` 固定为 `8`；
  - 各保留位和保留字节均为 `0`；
  - 路由条目总数不超过 `4`，且存在代理主路径条目；
  - `StationCount` 与 `128` 字节位图中的置位个数一致。
- `decode` 成功后，`ReceivedDiscoverListInfo` 直接指向输入缓冲区中的接收计数区；上行路由条目和位图会复制到结构体内部缓存。

## 与 1.0 版本对比

对比基准为 `doc/南网hplc+hrf双模模块1.0标准规范/2、计量自动化系统双模通信技术规范/《南方电网计量自动化系统双模通信技术规范 第2部分：无线通信接入协议》（报批稿）.pdf` 的 `8.1.4.11 发现列表报文`。

从报文整体布局看，`2.0` 版 `MMeDiscoverNodeList` 仍保持 `1.0` 的“固定 `42` 字节头 + 上行路由条目区 + 固定 `128` 字节发现站点位图区 + 接收发现列表信息区”骨架；主要变化集中在固定头内部对原保留位和保留字节的复用，未改变整帧的总体组织方式。

| 对比项 | 1.0 版 | 2.0 版 | 结论 |
| --- | --- | --- | --- |
| 章节归属 | `8.1.4.11`，归属于 `1.0` 双模无线通信接入协议 | `5.1.4.11`，归属于 `2.0` 数据链路层通信协议 | 规范迁移并重编号 |
| 固定头长度与外层骨架 | 固定头到字节 `41` 结束，后续依次为上行路由条目、`128` 字节发现站点位图和接收发现列表信息 | 与 `1.0` 相同 | 报文总长计算方式和可变区组织未变 |
| 字节 `12-15` 位域 | 字节 `12-14` 为 `31 bit` 保留，仅字节 `15` 的最高位表示“与代理站点通信成功率计算完成标志” | 在保持完成标志不变的前提下，字节 `12 bit[1:0]` 新增 `PLC 能力指示`，`bit[2]` 新增 `通道数`，其余 `bit[30:3]` 仍保留 | `2.0` 复用了原保留位，增加介质能力描述 |
| 字节 `37-41` 定义 | 字节 `37-41` 共 `40 bit` 全部为保留 | 字节 `37-40` 改为 `与代理的 RU_SNR 评估信息`，字节 `41` 保留 | `2.0` 在不扩长报文的前提下新增链路评估信息 |
| 路由条目、位图与接收计数区 | 上行路由条目仍为 `3` 字节/项，最大 `4` 项；发现站点位图仍固定 `128` 字节；接收发现列表信息仍按位图置位顺序逐项携带 `1` 字节计数 | 与 `1.0` 一致 | 主体业务数据区延续 `1.0` 定义 |

整体上，`2.0` 版 `MMeDiscoverNodeList` 属于在 `1.0` 基础上的兼容扩展，而不是报文格式重设计。

## 校准说明

- 原 PDF 为扫描版，`pdftotext` 抽取第 57-59 页几乎只有水印；本文件基于页面 OCR 结果和局部裁剪结果人工校准。
- `消息类型 0x0055`、固定头总长 `42` 字节以及字节 `12-41` 的字段顺序，使用 `h2-link-v2-sdk/SDK/src/spg/mac/mmsg.h` 中的 `MMeDiscoverNodeList_t` 做了交叉确认。
- SDK 中 `nr`、`nulroute`、`rtent_len` 字段命名和规范字段语义不完全直观；本实现按规范正文解释，分别落为：
  - `SentDiscoverListCount`
  - `UpLinkRouteEntryCount`
  - `ReceivedDiscoverListInfoEntryBitLength`
- `与代理的 RU_SNR 评估信息` 对应表 123 的 OCR 质量仍不足以可靠拆位，本次先保留为 `4` 字节原始域，后续如需进一步细化可再补充辅助解析接口。
